1. Регистрация/Авторизация
Описание
Данный раздел содержит информацию по процессу регистрации и авторизации пользователей. 
Регистрация
Регистрация на веб-сайте доступна для следующих ролей:

Клиент. Руководитель сети;
Менеджер региона;
Оператор.

Регистрация осуществляется с помощью электронной почты. Электронная почта при регистрации подтверждается кодом, направленным на электронную почту, указанную при регистрации.

Форма регистрации должна содержать следующие поля:

ФИО;
Электронная почта;
Номер телефона;
Пароль;
Подтверждение пароля;
Чекбокс "Согласие на обработку персональных данных (Пользовательское соглашение)".

Для следующих ролей должно быть предусмотрено отправление ссылки-приглашения. Данная ссылка направляется на электронную почту, при переходе по ссылке подтверждается уровень доступа пользователя и отображается страница с возможностью установить пароль для дальнейшей авторизации.

Клиент. Специалист;
Клиент. Управляющий точкой;
Инженер.

Критерии для валидации пароля: 

Минимум 8 символов;
Минимум одна буква любого регистра;
Минимум один специальный символ (! @ # $ % ^ & * ( ) - _ = + \ | [ ] { } ; : / ? . > <).
Авторизация
Авторизация осуществляется по электронной почте, указанной при регистрации, и паролю.
Администратор получает данные для авторизации от разработчиков проекта.
Восстановление доступа
Для восстановления доступа направляется ссылка для восстановления пароля на почту, указанную при регистрации.

2. Ролевая модель участников
Роли
Роль
Описание

Клиент. Специалист 
Пользователь, которому доступна подача заявки, просмотр предыдущих своих поданных заявок по оборудованиям, адресу и истории по конкретному оборудованию. 

Клиент. Управляющий точкой
Пользователь, которому доступна информация по всем заявкам для каждого оборудования по прикрепленному адресу. Выдает права для Клиент. Специалист.

Клиент. Руководитель сети
Пользователь, которому доступна вся информация по прикрепленным к нему адресам и каждому оборудованию. Видна общая статистика по оборудованию, поданным и решенным заявкам (возможна фильтрация по конкретному адресу).Выдает права для Клиент. Управляющий точкой и Клиент. Специалист.

Администратор
Главный администратор, управляющий входящими заявками: создает, проверяет, назначает, отслеживает, редактирует, закрывает.Выдает права для Клиента. Руководителя сети, Операторов, Менеджеров регионов.

Менеджер региона
Сотрудник сервисной службы по конкретному региону. Отображаются заявки только его региона и может создавать, проверять, назначать, отслеживать, редактировать, закрывать полученные заявки.Выдает права для Инженера своего региона

Инженер
Исполнитель заявок: выезжает к клиенту, решает технические проблемы, сообщает о ходе и завершении работ.Может создать заявку, если по прибытии на точку по заявке, есть еще неисправное оборудование

Оператор
Сотрудник, который уточняет детали каждой созданной заявки во время звонка. Публикует или отклоняет заявки. Инженер и Администратор региона увидит заявку только после уточнения деталей.Имеет возможность назначения заявки на Инженера

Алгоритм получения прав
Описание
В данном разделе описан процесс получения пользователями прав доступа.
Клиент
Предусмотрено 3 роли Клиента: 
Клиент. Специалист
Клиент. Управляющий точкой
Клиент. Руководитель сети

Клиент. Специалист
Сотрудник на адресе точки. Для получения доступа к системе получает ссылку-приглашение на электронную почту от Клиент. Управляющий точкой. Среди остальных ролей клиентов имеет наименьшее количество доступов.
Клиент. Управляющий точкой
Назначается Руководителем сети. Для получения доступа к системе получает ссылку-приглашение на электронную почту от Клиент. Руководителя сети. Может настраивать себе уведомления.
Клиент. Руководитель сети
Получает права от Администратора. Проходит процесс регистрации на веб-сайте. Имеет максимальное количество доступов среди Клиентов и может назначать или удалять Управляющих точкой и Специалистов. Настраивает уведомления себе.
Администратор
Получает права от команды разработки. Выдает права для Клиента. Руководителя сети, Операторов, Менеджеров регионов.
Менеджер региона
Получает права от Администратора. Проходит процесс регистрации на веб-сайте. Назначает Инженеров по своему региону. 
Оператор
Получает права от Администратора. Проходит процесс регистрации на веб-сайте. Не может управлять правами для других участников.
Инженер
Получает права от Менеджера региона. Для получения доступа к системе получает ссылку-приглашение на электронную почту от Менеджера региона.

Приоритетность заявки
№
Приоритет 
Описание

1
Высокий 
Поломка является критической, работа оборудования остановлена полностью до устранения неполадок, либо оборудование требует сложного ремонта или замены. Время решения заявки - 3-4 часа

2
Средний
Поломка оборудования незначительная, требуется прочистить или заменить элемент. Напитки реализуются, но с погрешностью.

3
Низкий
Оборудование может работать исправно, требуется плановое обслуживание. Не влияет на производство продукта.

Точки входа для открытия формы
В системе предусмотрено 2 точки входа для попадания в форму заявки: через QR-код или через веб-сайт.
QR-код
Пользователь с помощью мобильного устройства сканирует QR-код, расположенный на оборудовании, проходит авторизацию или регистрацию и получает доступ к заявке.
В этом кейсе в заявке будут предзаполнены следующие поля: Номер телефона, ФИО, Серийный номер, Модель, Вид оборудования, Юридическое лицо, Адрес, так как эти данные будут содержаться в коде. Все предзаполненные данные можно отредактировать.
Веб-сайт
Авторизованный пользователь заходит через веб-сайт и открывает заявку. В этом кейсе будут предзаполнены следующие поля: Номер телефона, ФИО.  Все предзаполненные данные можно отредактировать.
Отправление заявки
После нажатия на кнопку "Отправить заявку" пользователю отображается один из трех pop-up (заявка отправлена, ошибка, превышен лимит заявок).
Основной сценарий
Если в заявке заполнены все поля → кнопка "Отправить заявку" отображается активной и черным цветом. При нажатии отображается рop-up, содержащий следующий текст:
"Заявка успешно создана.
Номер заявки: №11111.
В ближайшее время с Вами свяжется наш оператор и уточнит все детали."

+ кнопка "Закрыть", которая перенаправляет пользователя на экран "Мои заявки".
Альтернативный сценарий 1
Если в форме заполнены не все обязательные поля, кнопка "Отправить заявку" неактивна и имеет серый цвет. Незаполненные поля подсвечиваются красным цветом.
Альтернативный сценарий 2
Если клиент не может отправить заявку на сервер по техническим причинам (например, слабый интернет), отображается рop-up, содержащий следующий текст:
Не удалось создать заявку. Пожалуйста, повторите позднее.

+ Кнопка закрыть, которая перенаправляет пользователя на экран "Мои заявки".
Альтернативный сценарий 3
Если клиент не может отправить заявку на сервер по превышению технического лимита отправки заявок (20 заявок в день с одного пользователя), то отображается pop-up, содержащий следующий текст:
Превышен лимит заявок. Пожалуйста, повторите позднее.

+ Кнопка закрыть, которая перенаправляет пользователя на экран "Мои заявки".
Взаимодействие с заявкой
После создания заявки, ей присваивается статус "Открыта" (opened). 
В списке заявок Клиент видит все доступные ему заявки. В заявке содержится вся заполненная информация, статус и поле с дополнительной информацией, которую добавляет Оператор.
Заявку можно отменить по соответствующей кнопке до статуса "Выполнено" (completed).
Ролевая модель. Клиент
Роли клиента
Клиент. Специалист - рядовой сотрудник на адресе, который следит за состоянием оборудования. 
Клиент. Управляющий точкой - управляющий по конкретному адресу с указанием всего оборудования по этой точке и информацией по поданным заявкам. Вывод статистики по адресу. 
Клиент. Руководитель сети - держатель нескольких адресов, с возможностью просматривать информацию по каждому адресу, оборудованию, заявки. Вывод общей статистики.

Функционал 
Описание

1
Создание заявки(1 релиз)
Специалист - пользователь, который заметил поломку кофемашины. Специалист находится в авторизованной зоне.Специалист может создать заявку двумя способами:
Через веб-сайт
Через QR-код

2
Просмотр заявки(1 релиз)
Специалист может просмотреть подробную информацию по своим заявкам. Редактировать заявку нельзя. 

3
Просмотр всех заявок(1 релиз)
Специалисту доступен просмотр созданных им заявок. Все заявки располагаются в хронологическом порядке. Есть фильтрация по статусу, оборудованию.Заявки не в статусе “Выполнено” находятся в начале списка.

По кнопке "Приступить к работе" у заявки со статуса "enroute" - инженер в пути, меняется на в "in_work" - заявка в работе. 
По кнопке "Готово" у заявки меняется статус с "in_work"- в работе, на "completed"- завершена.
По кнопке "Есть проблемы" у заявки меняется статус с "in_work" - в работе, на "on_hold" - приостановлена. Требуется обязательно оставить комментарий. Далее заявка переходит Менеджеру региону/Администратору.
При нажатии на каждую из кнопок при смене статуса, система считает время выполнения по каждому статусу для дальнейшего отображения аналитики для Администратора/Менеджера региона

6
Приостановление заявки(1 релиз)
При возникновении форс-мажорных ситуаций Инженер может нажать на кнопку "Есть проблемы". У заявки меняется статус на "on_hold" - приостановлена. Требуется обязательно оставить комментарий с причиной приостановления заявки. Далее заявка переходит Менеджеру региона/Администратору/Оператору, который может назначить задачу другому Инженеру.

7
Рекомендация Инженера(2 релиз)
Инженер может оставить рекомендацию по оборудованию на точке после выполнения заявки, которая будет отображаться у всех ролей. Можно просмотреть рекомендации по конкретной точке.

8
Комментарий Инженера(2 релиз)
Инженер заполняет форму о проделанной работе для дальнейшего формирования счета на оплату. Форма состоит из полей "Выполненные работы", "Используемые запчасти" в виде раскрывающихся списков. Не отображается у Клиента. Отображается в виде текстового комментария Администратору, Менеджеру региона, Инженеру.

9
Управление статьями / FAQ(3 релиз)
Инженер имеет возможность чтения статей. Не может их добавлять/изменять/удалять.

10
Уведомления(1 релиз)
Инженер получает уведомления на электронную почту о назначенной на него новой заявке. 

11
Профиль(1 релиз)
В Профиле пользователь видит информацию о себе: 
ФИО;
Фотография;
Электронная почта;
Номер телефона;
Регион, к которому привязан Инженер.
При отсутствии фотографии должно отображаться уведомление-напоминание, чтобы Инженер поставил фотографию в Профиле.

Цель
Обеспечить инженера актуальной информацией по списку и приоритету выполнения по назначенной на него заявке для решения проблемы клиента.
Общие требования

Инженер может открыть карточку заявки для просмотра деталей.
Дата исполнения и время прибытия рассчитывается автоматически или назначается Менеджером региона/Администратором.

Инженер изменяет статус заявки на "Выполнена" после выполнения работы.

9
Назначение инженера(1 релиз)
Менеджер региона просматривает все заявки в статусе "Открыта" для назначения Инженера и может выбрать доступного Инженера из списка и передать в работу. Также Менеджер региона имеет возможность поменять Инженера по внутренним причинам с помощью текстового поля.

10
Ответ на отзыв(2 релиз)
Менеджер региона может один раз оставить обратный ответ на отзыв пользователя после статуса заявки "Выполнено". 

11
Получение уведомлений ( релиз)
Менеджер региона получает уведомления только о поступлении опубликованных заявок после работы назначения Оператором. Также предусмотрены отложенные уведомления-напоминания на определенный день для назначения заявки.

12
Управления статьями/ FAQ(2 релиз)
Менеджер региона может читать все имеющиеся на платформе статьи. Отсутствует возможность редактирования, добавления и удаления.

13
Доступ к аналитике(2 релиз)
Менеджер региона имеет доступ к просмотру аналитики только по своему региону, населенным пунктам, точкам с оборудованием.

14
Отчеты(2 релиз)
Менеджер региона имеет доступ к динамическим отчетам о выездах Инженеров. Содержит поля:
Регион;
Населенный пункт;
Дата работы;
ФИО Инженера;
Наименование контрагента;
Выполненные работы;
Используемые запчасти;
Стоимость запчасти;
Стоимость выезда Инженера;
Время инженера в пути, ч;
Расстояние, км;
Расстояние, руб;
Время решения заявки (в работе), ч;
Стоимость решения заявки, руб;
Итоговая стоимость.
Предусмотрена возможность экспортировать отчет в формате Excel.

15
Управление пользователями(1 релиз)
Осуществляет выдачу прав Инженеру.

16
Просмотр точек
Точки отсортированы по городам, относящиеся к региону. Города выводятся в алфавитном порядке (А-Я). Администратор региона имеет возможность просмотра, создания, редактирования и удаления точек.

17
Список оборудования(1 релиз)
Менеджер региона имеет доступ к просмотру, созданию, редактированию и удалению полного списка имеющегося оборудования в его регионе

18
Генерация ссылки-приглашения для подтверждения пользователя
(2 релиз)
Менеджеру региона доступна возможность генерации ссылки-приглашения для подтверждения пользователей. Обновляется по запросу пользователя по нажатию кнопки. 
Кнопка для генерации ссылки располагается в Профиле. Ссылка направляется на указанную при регистрации электронную почту пользователя.

19
Профиль(1 релиз)
В Профиле пользователь видит информацию о себе:
ФИО;
Электронная почта;
Номер телефона;
Регион.
Доступна кнопка для генерации ссылки-приглашения для подтверждения пользователя (2 релиз). 


Общие требования
Администратор/Менеджер региона управляет заявками, Инженерами и настройками системы.
Администратор может:
Просматривать все заявки (с фильтрацией по статусам и регионам) (в случае Менеджера региона - просмотр заявок только своего региона).
Вручную назначать заявки Инженерам.
Изменять приоритет и тип проблемы в заявке.
Управлять справочниками (типы проблем, регионы, оборудование) (только Администратор).
Получать уведомления о просроченных заявках.
Инициирующие события
Администратор/Менеджер региона открывает интерфейс управления заявками.
Система отправляет уведомление о заявке, которую никто не взял в работу.
Заключительные события
✅ Успешное завершение:
Заявка назначена Инженеру.
Приоритет/тип проблемы изменены.
❌ Отмена:
Действие отменено (например, не выбран инженер).
Use Case 1: Администратор/Менеджер региона просматривает список заявок
Основной сценарий:
Администратор/Менеджер региона открывает раздел "Заявки".
Система отображает заявки с фильтрами:
Статус;
Регион;
Приоритет;
Дата создания.
Каждая карточка содержит:
Номер заявки;
Описание проблемы клиента;
Вид оборудования;
Комментарий Оператора;
Комментарий Инженера;
Дата создания;
Дата последнего изменения;
Приоритет;
Тип проблемы;
Данные о клиенте;
Данные об исполнителе;
Оценка клиента;
Комментарий к оценке;
История изменения статусов и время в каждом из статусов.
Администратор/Менеджер региона может:
Открыть детали заявки.
Применить дополнительные фильтры.
Use Case 2: Администратор/Менеджер региона вручную назначает заявку Инженеру
Условия:
Заявка в статусе OPEN более N часов.
Заявка не назначена никому из Инженеров.
Основной сценарий:
Администратор/Менеджер региона видит агрегированное уведомление о просроченной заявке.
Открывает карточку заявки → видит подробную информацию.
Нажимает "Назначить инженера".
Система предлагает:
Список инженеров (фильтр по региону).
Поле "Дата исполнения" (обязательное).
Поле "Время прибытия" (опциональное).
Администратор/Менеджер региона выбирает Инженера, заполняет дату → подтверждает.
Система:
Меняет статус заявки на ASSIGNED.
Назначает Инженера.
Отправляет уведомление Инженеру.
Use Case 3: Администратор/Менеджер региона изменяет приоритет/тип проблемы
Основной сценарий:
Администратор/Менеджер региона открывает карточку заявки.
Нажимает "Изменить приоритет" или "Изменить тип проблемы".
Выбирает новое значение из списка:
Приоритет: Низкий / Средний / Высокий.
Тип проблемы: Техническая / Рецептурная / Другое.
Подтверждает изменение.
Система обновляет данные заявки.
Use Case 4: Администратор управляет справочниками
Основной сценарий:
Администратор открывает раздел "Справочники".
Выбирает категорию:
Типы проблем (добавить/изменить/удалить).
Регионы (добавить/изменить).
Оборудование (модели, серийные номера).
Вносит изменения → сохраняет.
Система обновляет данные, изменения применяются для новых заявок.
Use Case 5: Администратор/Менеджер региона просматривает аналитику
Основной сценарий:
Администратор/Менеджер региона открывает раздел "Аналитика".
Система отображает Аналитику для Администратора/Менеджера региона. Показатели находятся в разделе "Аналитика".
Администратор/Менеджер региона может:
Применить фильтры по статусу выполнения, периоду, региону (для Администратора), адресу, оборудованию.
Экспортировать отчет.
Use Case 6: Администратор генерирует QR-код
Основной сценарий:
Администратор открывает раздел "Генерация QR-кода".
Нажимает "Сгенерировать код".
Система генерирует QR-код и выводит его на экран.
Пользователь нажимает "Скачать" и скачивает QR-код на устройство в одном из форматах: png, jpg, jpeg.
Файл с изображением скачивается на устройство.
